## 0. 核心铁律（违反任何一条，立即停止并回滚）

| # | 铁律 | 解释 |
|---|------|------|
| **0.1** | **先读后写，禁止盲写** | 修改任何文件前，必须先用 Read 工具读取该文件当前内容。不看代码就改 = 灾难 |
| **0.2** | **一次只做一件事** | 每次对话只实现/修改一个模块、一个类或一个函数。禁止一次性生成多个不相关的类 |
| **0.3** | **不破坏已有功能** | 修改后，已有功能必须照常运行。如果改动会影响其他模块，必须先列出影响范围 |
| **0.4** | **遵循已有模式，不发明新模式** | 项目中已有命名、结构、风格的，严格遵循。不要用"我觉得更好"的理由改别人的代码风格 |
| **0.5** | **能编译通过才算完成** | 每次代码生成后，确保项目可编译。不允许留下语法错误让用户自己修 |

## 2. 代码编写规范

### 2.1 文件结构规范

**每个类 = 两个文件（一个 .h + 一个 .cpp），没有例外。**

```
include/entity/Plant.h        ← 类声明
project/src/entity/Plant.cpp  ← 类实现
```

**头文件模板：**

```cpp
#ifndef PVZ_PLANT_H
#define PVZ_PLANT_H

// 标准库 include
#include <string>
// 项目内 include
#include "component/Animation.h"

// 前向声明（能用前向声明就不用 #include）
class Bullet;

class Plant {
public:
    // ---- 构造/析构 ----
    Plant(/* 参数 */);
    virtual ~Plant();

    // ---- 禁止拷贝 ----
    Plant(const Plant&) = delete;
    Plant& operator=(const Plant&) = delete;

    // ---- 公共接口 ----
    virtual void Update(float dt);
    virtual void Render();

    // ---- Getter ----
    int GetRow() const { return row; }
    int GetCol() const { return col; }

protected:
    // ---- 子类可访问的数据 ----
    int row, col;
    int health;
    int maxHealth;
    Animation anim;
};

#endif // PVZ_PLANT_H
```

**源文件模板：**

```cpp
#include "entity/Plant.h"
#include "utils/Constants.h"   // 常量定义

Plant::Plant(/* 参数 */)
    : row(0), col(0), health(0), maxHealth(0) {
}

Plant::~Plant() = default;

void Plant::Update(float dt) {
    // 逻辑实现
}

void Plant::Render() {
    // 绘制实现
}
```

### 2.2 命名规范（强制）

| 元素 | 规则 | 示例 |
|------|------|------|
| **类名** | PascalCase | `PeaShooter`, `CollisionManager`, `GameplayScene` |
| **函数/方法** | PascalCase | `Update()`, `GetHealth()`, `OnPlantClicked()` |
| **成员变量** | camelCase | `currentHealth`, `attackSpeed`, `isAlive` |
| **局部变量** | camelCase | `deltaTime`, `plantCount`, `mousePos` |
| **常量/枚举** | UPPER_CASE | `MAX_ROW`, `SUN_PRICE`, `DEFAULT_FPS` |
| **宏定义** | UPPER_CASE + 模块前缀 | `PVZ_PLANT_H`, `IMG_PLANT(...)` |
| **文件名** | 与类名一致 | `PeaShooter.h`, `PeaShooter.cpp` |

**命名禁止事项：**

- ❌ 单字母变量（循环索引 `i`, `j` 除外）
- ❌ 拼音命名
- ❌ 无意义缩写（`mgr` → `Manager`, `sz` → 不用匈牙利前缀）
- ❌ 数字后缀（`plant1`, `plant2`）
- ❌ 布尔变量不用 `b` 前缀，直接用 `is/has/can` 开头

### 2.3 代码组织规范

**函数内组织（从上到下）：**

```cpp
void SomeClass::Update(float dt) {
    // 1. 边界检查与早返回
    if (!isAlive) return;

    // 2. 状态更新
    attackTimer += dt;

    // 3. 条件触发
    if (attackTimer >= attackInterval) {
        Shoot();
        attackTimer = 0.0f;
    }

    // 4. 子组件更新
    anim.Update(dt);
}
```

**类内声明顺序（.h 文件中）：**

```cpp
class Example {
public:
    // 1. 构造/析构
    // 2. 删除的拷贝/移动
    // 3. 公共方法（按逻辑分组，每组之间空一行）
    // 4. Getter/Setter

protected:
    // 5. 子类可访问的方法
    // 6. 子类可访问的数据

private:
    // 7. 私有方法
    // 8. 私有数据
};
```

### 2.5 Update / Render 分离（强制）

```cpp
// 错误 ❌ —— 在 Render 中修改状态
void Plant::Render() {
    health -= 1;                // 禁止！Render 只能绘制
    if (++frameCount > 10) {    // 禁止！Render 不能有状态变更
        frameCount = 0;
    }
    putimage(x, y, &sprite);
}

// 正确 ✅
void Plant::Update(float dt) {
    health -= static_cast<int>(damagePerSec * dt);
    anim.Update(dt);      // 状态变更在这里
}

void Plant::Render() {
    anim.Render(x, y);    // 只绘制
}
```

**检测标准：** 如果注释掉 `Render()` 函数体，游戏逻辑应该完全不受影响。

### 2.6 内存与资源管理

```cpp
// 必须使用智能指针管理实体容器
std::vector<std::unique_ptr<Plant>> plants;
std::vector<std::unique_ptr<Zombie>> zombies;
std::vector<std::unique_ptr<Bullet>> bullets;

// 禁止裸指针管理动态对象（管理器类如 Game/SceneManager 除外）
// 错误 ❌
Plant* plant = new SunFlower(...);
plants.push_back(plant);

// 正确 ✅
plants.push_back(std::make_unique<SunFlower>(...));

// 删除对象时必须安全遍历
// 错误 ❌ —— 迭代器失效
for (auto it = zombies.begin(); it != zombies.end(); ++it) {
    if ((*it)->GetHealth() <= 0) {
        zombies.erase(it);  // 危险！
    }
}

// 正确 ✅ —— 使用 erase-remove 或反向遍历
zombies.erase(
    std::remove_if(zombies.begin(), zombies.end(),
        [](const auto& z) { return z->GetHealth() <= 0; }),
    zombies.end()
);
```

## 4. 场景开发规范

### 4.1 场景开发流程

每个场景必须按以下顺序开发：

```
1. 创建头文件 include/scene/XxxScene.h
   └── 声明类，继承 Scene，声明 Update() 和 Render() 为 override

2. 创建源文件 project/src/scene/XxxScene.cpp
   └── 实现 OnEnter()、OnExit()、Update()、Render()

3. 在 SceneManager 中注册：
   └── sceneManager->RegisterScene(SceneID::Xxx, std::make_unique<XxxScene>());

4. 添加场景切换触发条件（按钮/按键）
```

### 4.2 场景基类接口（Scene）

```cpp
class Scene {
public:
    virtual ~Scene() = default;

    // 场景激活时调用（加载资源、初始化状态）
    virtual void OnEnter() = 0;

    // 场景退出时调用（释放场景特有资源、保存状态）
    virtual void OnExit() = 0;

    // 每帧调用（逻辑更新）—— 纯虚函数，必须实现
    virtual void Update(float dt) = 0;

    // 每帧调用（绘制）—— 纯虚函数，必须实现
    virtual void Render() = 0;
};
```


### 5.2 新增僵尸检查清单

```
□ 继承 Zombie 基类
□ 实现虚函数 Update(float dt)
□ 实现虚函数 Render()
□ 构造函数中初始化所有 Zombie 基类属性

```

### 5.3 实体状态机实现模式

```cpp
// 僵尸状态机示例
enum class ZombieState {
    Walking,
    Attacking,
    Dead
};

void Zombie::Update(float dt) {
    switch (state) {
    case ZombieState::Walking:
        x -= speed * dt;                       // 向左移动
        if (CheckPlantCollision()) {
            state = ZombieState::Attacking;    // 遇到植物
        }
        if (x <= houseBoundary) {
            // 触发游戏失败（通过回调或标记通知场景）
            reachedHouse = true;
        }
        anim.SetAnimation("walk");
        break;

    case ZombieState::Attacking:
        attackTimer += dt;
        if (attackTimer >= attackInterval) {
            targetPlant->TakeDamage(attackPower);
            attackTimer = 0.0f;
        }
        if (!targetPlant || !targetPlant->IsAlive()) {
            state = ZombieState::Walking;      // 植物死了，继续走
        }
        anim.SetAnimation("attack");
        break;

    case ZombieState::Dead:
        anim.SetAnimation("death");
        if (anim.IsFinished()) {
            isAlive = false;                    // 等待清理
        }
        break;
    }

    anim.Update(dt);
}
```

## 7. 禁止事项清单（零容忍）

| # | 禁止行为 | 后果 | 正确做法 |
|---|---------|------|---------|
| 1 | 不读文件直接修改 | 破坏已有逻辑 | 先 `Read`，再 `Edit` |
| 2 | 一次性生成 > 200 行代码 | 难审查、难调试 | 分多次，每次 < 150 行 |
| 3 | 在 `Render()` 中修改成员变量 | Update/Render 混乱 | 所有状态变更放 `Update()` |
| 4 | 使用裸指针管理堆对象 | 内存泄漏 | 使用 `std::unique_ptr` |
| 5 | 硬编码数值 | 不可维护 | 使用 `Constants.h` 中常量 |
| 6 | 硬编码资源路径 | 不可移植 | 使用 ResourceManager 宏 |
| 7 | 修改已有类的 public 接口 | 破坏兼容性 | 新增接口，不改已有接口 |
| 8 | 删除他人代码而不说明 | 功能丢失 | 注释并说明原因 |
| 9 | 使用 `using namespace std` 在头文件 | 命名空间污染 | 在 .cpp 中使用或完全限定 |
| 10 | 提交无法编译的代码 | 阻塞整个项目 | 每次修改后检查编译 |
| 11 | 不使用 `override` 关键字 | 虚函数签名错误难发现 | 所有重写虚函数加 `override` |
| 12 | `#include` 不需要的头文件 | 编译变慢、耦合增加 | 仅 include 实际使用的 |
| 13 | 在 .h 中写函数实现（模板除外） | 编译依赖爆炸 | 实现放 .cpp |
| 14 | 使用 `NULL` 或 `0` 表示空指针 | 类型不安全 | 使用 `nullptr` |
| 15 | 修改代码时不保持缩进一致 | 代码可读性下降 | 统一使用 4 空格缩进，不用 Tab |
| 16 | 在 `.h` 文件中 `using namespace std` | 污染所有包含该头文件的编译单元 | 在 `.cpp` 中使用 |
| 17 | 全局变量 | 不可控的状态 | 通过单例管理器访问 |

## 8. 编译与验证规范

### 8.1 每次修改后的必要检查

```
□ 项目编译通过（0 error, 0 warning）
□ 新增的 .h / .cpp 文件已添加到 PVZ2677.vcxproj
□ 新增的 .h / .cpp 文件已添加到 PVZ2677.vcxproj.filters
□ 没有引入新的全局变量
□ 没有在 .h 中增加不必要的 #include
□ 使用的主逻辑都在 Update() 中，绘制都在 Render() 中
```

### 8.2 VS 项目文件维护

每次新增文件，必须同步更新两个 VS 配置文件：

**在 `PVZ2677.vcxproj` 中添加：**

```xml
<!-- 在 <ItemGroup> 中添加 .h -->
<ClInclude Include="include\entity\NewPlant.h" />

<!-- 在 <ItemGroup> 中添加 .cpp -->
<ClCompile Include="project\src\entity\NewPlant.cpp" />
```

**在 `PVZ2677.vcxproj.filters` 中添加：**

```xml
<!-- .h -->
<ClInclude Include="include\entity\NewPlant.h">
  <Filter>entity</Filter>
</ClInclude>

<!-- .cpp -->
<ClCompile Include="project\src\entity\NewPlant.cpp">
  <Filter>entity</Filter>
</ClCompile>
```

### 8.3 增量修改原则

```
修改优先级：
1. 新增 .h + .cpp（最安全，不影响现有代码）
2. 在已有 .cpp 中追加函数实现（较安全）
3. 在已有 .h 中追加声明（较安全，注意 include guard 内）
4. 修改已有函数实现（中等风险，必须理解原逻辑）
5. 修改已有 .h 声明 / 修改类接口（高风险，必须列出所有影响点）
```

## 9. 常见 AI 错误及预防

### 9.1 经典错误模式与预防

| 常见错误 | 为什么发生 | 预防措施 |
|---------|-----------|---------|
| 把 game loop 写在 main 里几百行 | AI 偷懒，不想拆类 | 必须用 Game 类封装主循环 |
| 所有植物逻辑写在一个大 switch | 觉得"简单、快" | 必须用多态，每种植物一个类 |
| 在 Render 里碰撞检测、伤害计算 | 不区分逻辑层和表现层 | 记住铁律：Render 只画图 |
| 裸指针到处飞，内存泄漏 | 不会用 smart pointer | 强制使用 unique_ptr |
| 改了 A 类的接口，B 类编译报错 | 没看依赖关系 | 改接口前查阅架构文档依赖图 |
| 头文件 include 了一大堆，循环依赖 | 没用前向声明 | .h 文件优先前向声明 |
| 数值写死，调平衡要改十几处 | 图省事 | 所有数值进 Constants.h |
| 资源不存在程序直接崩 | 没做空指针检查 | 所有 GetImage 返回值检查 + 占位 |
| 代码组织混乱，一会上定义下实现 | 随性写 | 严格按本文档第 2.3 节顺序 |
| 写了几个类但不加到 vcxproj | 忘记项目配置 | 按 8.2 节检查清单执行 |

### 9.2 自检问题清单

AI 在每次回复代码前，必须自问：

1. 我读过这个文件了吗？
2. 这段代码会影响哪些其他模块？
3. 有没有硬编码的数值？
4. 有没有硬编码的路径？
5. Update 和 Render 是否严格分离了？
6. 新增的 .h / .cpp 文件是否都要加到 vcxproj？
7. 如果是新增类，是否已在架构设计文档中定义？
8. 是否使用了智能指针而不是裸指针？

## 10. AI 与用户协作规范

### 10.1 沟通格式

```
AI 发起修改时必须使用以下格式：

📋 **任务：** [做什么]
📁 **文件：** [涉及文件列表]
⚠️ **影响：** [影响范围]
💡 **方案：** [实现思路，2-3 句话]

是否继续？

用户回复"继续"或"OK"后，AI 开始编码。
编码完成后：

✅ **完成：** [做了什么]
🔨 **新增文件：** [文件名列表]
✏️ **修改文件：** [文件名列表]
📝 **注意：** [需要用户关注的事项]
```

### 10.2 用户反馈的处理

| 用户反馈 | AI 响应 |
|---------|--------|
| "编译报错了" | 读取报错信息 → 定位文件 → 读文件 → 修复 → 确保编译通过 |
| "这个逻辑不对" | 确认需求 → 读功能文档 → 列出理解 → 确认后再改 |
| "为什么这里要这样写？" | 解释设计理由，引用架构文档或规范文档 |
| "能不能一起把 X 也做了？" | 回答：建议先完成当前任务再处理 X，避免改动过大 |
| "不要用 unique_ptr" | 接受，但提醒可能的内存风险 |

### 10.3 未知情况的处理

当 AI 遇到以下情况时，必须询问用户，不得自行决定：

1. 需要使用的功能在架构文档中没定义
2. 实现方案有 A/B 两种合理选择
3. 改一个地方会影响 3 个以上其他模块
4. 发现代码中有看似 bug 的东西，但不确认
5. 需要引入新的第三方库
